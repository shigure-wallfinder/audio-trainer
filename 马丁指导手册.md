# 马丁指导手册

> 用于指导 AI Agent 与 Martin 协作
> 基于历史对话总结

---

## 一、沟通风格

### 1.1 说话方式

| 特点 | 描述 |
|------|------|
| **语言** | 中文优先 |
| **语气** | 直接、简洁、务实 |
| **反馈** | 具体、有建设性 |
| **偏好** | 结果导向，关注产出 |

### 1.2 指令特点

- **明确任务**：会给具体角色分配任务（Developer/Designer/Tester）
- **优先级**：会说明优先级（P0/P1/P2）
- **范围界定**：会说明哪些要做、哪些不做
- **限制条件**：会提供必要信息（如密码）

---

## 二、工作习惯

### 2.1 代码管理

| 事项 | 要求 |
|------|------|
| **GitHub** | 代码必须提交到 GitHub |
| **敏感信息** | 密码/API Key 不能上传 |
| **构建验证** | 每次代码更新后 build 检查 |
| **测试** | 重视测试报告 |

### 2.2 项目管理

- 定期汇报进度
- 对接检查（Designer ↔ Developer）
- 测试驱动修复（Tester → Developer）
- 分阶段推进（设计→开发→测试→推广）

---

## 三、偏好与禁忌

### 3.1 喜欢

- ✅ 结构化信息（表格、列表）
- ✅ 明确的进度报告
- ✅ 问题及时反馈
- ✅ 代码质量重视
- ✅ 测试驱动

### 3.2 不喜欢

- ❌ 长篇大论的思考过程
- ❌ 未经测试的代码
- ❌ 敏感信息上传
- ❌ 不明确的指令

---

## 四、常见指令模式

| 指令类型 | 示例 |
|----------|------|
| **进度查看** | "查看进度" |
| **任务分配** | "Developer，做xxx" |
| **测试请求** | "Tester，测试xxx" |
| **对接检查** | "Designer和Developer对接" |
| **代码修复** | "Developer，修复问题" |
| **提交上传** | "提交到GitHub" |

---

## 五、关键约束

### 5.1 安全要求

- 数据库密码本地保存，不上传 GitHub
- 使用环境变量处理敏感配置
- API Key 等需要用户自己配置

### 5.2 质量要求

- 代码必须能通过构建
- 测试通过后才能进入下一阶段
- 对接检查发现的问题必须修复

---

## 六、工作流程

```
1. Designer ↔ Developer 对接 → 产 出企划书
2. Developer 开发 → 代码 + 文档
3. Tester 测试 → 测试报告
4. 反馈修复 → 循环直到通过
5. 推进到下一阶段
```

---

## 七、重要提示

1. **不要上传敏感信息**：密码、API Key、密钥等
2. **先 build 再提交**：确保代码能编译通过
3. **测试驱动**：Tester 发现问题 → Developer 修复
4. **及时同步**：进度、问题要及时汇报
